# Code & Naming Standards

We use modified Hungarian notation across all of the C++ codebases, and it's not optional in review, because `clang_check` enforces the formatting and the reviewers enforce the naming. The canonical source for all of this is each repo's `CONTRIBUTING.md` and the [C++ Style Guide](../reference/links), and the tables below are reproduced from it.

The reason we bother with it is readability. The prefixes let you look at any variable and immediately know what type it is and whether it's a member or a local variable. On a codebase that a rotating cast of students works on, that saves a lot of time, because you can read unfamiliar code without constantly jumping back to the declarations.

## Names, macros, and documentation

- Functions, classes, and structs use PascalCase: `InitializeSensor()`, `class StateMachineHandler`.
- Macros are `ALL_CAPS_WITH_UNDERSCORES`, e.g. `#define AUTONOMY_MAJOR_VERSION 24`.
- Every file, class, method, and function gets a Doxygen comment block. In the dev container, type `/**` then Enter to scaffold one. Docs build from these automatically on each push to `development`.
- Formatting is owned by `.clang-format`. Run format-on-save and don't fight it.

## In-code task markers

We tag work in comments with keywords the TODO extension can compile into a list:

`// TODO:`, `// NOTE:`, `// BUG:`, `// HACK:`, `// FIXME:`, `// LEAD:`, `// @todo`, `// ISSUE NOTE:`, `// TEST:`, plus checklist-style `// [ ]` and `// [x]`.

## The C# side

The Blazor Basestation repo follows normal .NET conventions (PascalCase, an `.editorconfig`), not Hungarian. Match whatever the file you're editing already does.
