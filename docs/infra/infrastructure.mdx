# Infrastructure

## AutoPC (SDELC)

A dedicated **Ubuntu** machine in the SDELC (Student Design & Experiential Learning Center). Software members use it to develop on, and it occasionally hosts services for the team. It's also the **x64 self-hosted GitHub Actions runner** (`[self-hosted, linux, X64]`), so it runs the unit/integration tests, the AMD64 package rebuilds, the `autonomy-jammy` image build, and the release job (see [CI/CD](../foundations/github#what-you-have-to-set-up-the-two-runners)). The matching ARM64 runner is a Jetson.

:::tip[ACTION]
Confirm the runner is registered and online (Org → Settings → Actions → Runners) and set to start on boot, and that you have physical + remote access. If tests or image builds sit "queued" forever, this box (or the Jetson runner) is suspect #1.
:::

## CCF PC (Simulator / GitLab)

This is the PC that self-hosts the [Simulator's GitLab](../subteams/simulator). The reason it lives where it does is bandwidth: it needs to sit somewhere with a **symmetric gigabit connection**, and the CCF (down the street near the alumni house) was the only place on campus that had one.

The catch is that this machine has to live wherever a team member can physically host it. Over the summer, **Brendan Westley** took the PC and is connecting it from his place in **St. Louis**. So "where is the sim GitLab" is a question whose answer moves around, always confirm who currently has the box.

:::tip[ACTION]
Find out who currently holds the CCF PC, get GitLab owner rights, clone the sim project, document its build/run, and verify GitLab backups. Plan for it to need a long-term home with good upload bandwidth.
:::

## Handoff checklist

For each of VaultWarden, the AutoPC, and the CCF PC, write down where it physically is, how to reach it remotely, who has admin on it, and whether backups exist and have actually been tested.
